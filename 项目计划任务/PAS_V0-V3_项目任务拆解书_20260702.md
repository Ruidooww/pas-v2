## 1. 项目重启原则

### 1.1 当前决策

PAS 项目重启后，不再以“一次性建设完整 PAS”为首期目标，而是按 V0/V1/V2/V3 分阶段交付：

- V0：下周结束前，全公司可用“知识库问答 + 客户分析 + 一键方案 + 多格式交付物（docx/pptx/xlsx）”。
- V1：把 V0 从可用提升到稳定可运营，补齐质量评估、权限、模板、反馈闭环。
- V2：扩展到商机、会议、售后、合同、渠道等业务闭环。
- V3：再考虑多渠道、经营驾驶舱、Agent 编排、CIP 深化、商业化/多租户能力。

### 1.2 技术边界

- RAG 不自研，知识库底座外接 RAGFlow。
- CRM 不自建，外接现有 CRM 接口，PAS 只读取客户和商机上下文。
- PAS 自有运行容器限定为 `frontend`、`backend`、`postgres`、`redis` 四类；RAGFlow、CRM、LLM、外部对象存储等作为外部能力接入，不计入 PAS 自有容器。
- 后端所有模块、后台任务、队列 processor、定时任务统一封装在 `backend` 容器内，V0-V3 基线不拆独立业务 worker 容器。
- 文件能力统一封装在 `FilesModule`；`MinIO Client` 可作为 S3-compatible SDK 放在 `backend` 依赖内，但 V0 默认使用 `local driver + backend volume`，不内嵌 `MinIO Server`。
- docx/pptx/xlsx 不走 AI 自由设计，采用“公司标准模板 + 占位符/固定结构 + 内容填充 + 自动检查”。V0 使用真模板填充链路，不用代码从零重画公司模板。
- 方案生成遵循“不创作只组装”：模板、知识块、规则为主，LLM 只做结构化、提炼、过渡表达和少量定制化内容。
- AI 输出必须保留人工确认机制，不能自动外发关键材料。

### 1.3 首期成败标准

V0 的目标不是功能多，而是让销售、售前能在实际工作里马上用起来：

- 能问知识库。
- 能选择 CRM 客户或商机。
- 能生成客户情况分析。
- 能生成 IP-Guard 售前方案草稿。
- 能生成符合公司标准模板的 docx、pptx 和 xlsx。
- 能看到引用来源。
- 能下载、编辑、人工确认后使用。

### 1.4 旧项目资产处理原则

当前重启策略采用“新项目重建 + RAGFlow 资产迁移复用”：

- **复用：RAGFlow 相关资产。** 包括 RAGFlow 部署方式、可用知识库资料、可用 KB/dataset 配置、API 调用经验、检索参数、rerank 配置、引用映射经验、知识库质量回归问题集。
- **不复用：旧 E0-E5 工程代码。** 旧项目中的前后端代码、Next.js 页面、旧 NestJS 模块、旧 Prisma schema/migration、旧 worker、旧权限实现、旧部署脚本不作为新项目实现基线。
- **只参考：旧 E0-E5 的经验和接口边界。** 可以阅读旧代码来避免重复踩坑，但不能把旧模块原样迁入新项目，也不能继续沿用旧 E0-E5 任务编号驱动研发。
- **RAGFlow 不直接上线旧库。** 旧库只作为迁移来源，V0 必须建立新 V0 知识库空间或明确标记的 V0 dataset，并通过资料清单、解析检查、50 个真实问题回归后再进入全员使用。
- **PAS 重新建立工程边界。** 新项目按 `Vite + React + Ant Design + NestJS API + PostgreSQL + Prisma + Redis/BullMQ + FilesModule + Nginx 静态 SPA` 建设，PAS 自有容器只保留 `frontend/backend/postgres/redis` 四类。

迁移口径：

```text
旧 PAS 工程
  -> 只迁移 RAGFlow 资产和经验
  -> 不迁移 E0-E5 应用代码
  -> 新 PAS 按 V0-V3 任务书重新实现
```

### 1.5 Agent 开发周期预估

以下周期按“Agent 主导开发 + 项目负责人及时提供资料/API/模板 + 人工验收配合”的口径估算。

| 范围 | 单主 Agent 预估 | 多 Agent 并行预估 | 关键前提 |
|---|---:|---:|---|
| V0 战役版 | 7-10 个自然日 | 5-7 个自然日 | RAGFlow 可用、V0 dataset 首批资料到位、CRM 至少有 mock/测试接口、docx/pptx/xlsx 模板到位 |
| V1 稳定 MVP | 2-3 周 | 10-15 个工作日 | V0 已有真实反馈，能持续跑质量回归，知识块审核负责人到位 |
| V2 业务闭环版 | 4-6 周 | 3-4 周 | CRM/会议/合同/售后/渠道接口和业务负责人到位 |
| V3 平台化版本 | 6-10 周 | 4-6 周 | 多渠道、驾驶舱、Agent 编排需要稳定数据基础 |
| V0-V3 全量 | 3-5 个月 | 2.5-4 个月 | 需求边界不频繁反复，验收标准稳定 |

Agent 执行风险：

- 多 Agent 并行能压缩时间，但必须按模块拆 issue，并设置统一架构审查人，否则容易接口不一致。
- RAGFlow 解析质量、引用质量和模板素材是 V0 最大不确定因素；如果资料和模板不完整，V0 可能额外增加 2-4 天。

### 2.2 V0 验收目标：全公司可用战役版

V0 的验收目标是“可实际使用”，不是“功能完整”。验收时必须用真实客户、真实资料、真实销售/售前问题验证。

#### 业务验收

- 销售能选择或搜索一个 CRM 客户/商机。
- 系统能基于 CRM 数据生成客户情况分析。
- 售前能围绕该客户生成 IP-Guard 方案草稿。
- 售前能基于方案草稿生成 docx、pptx 和 xlsx。
- 管理员能维护第一批知识资料。
- 全公司用户能通过 Web 入口问知识库。
- 飞书 Bot 问知识库入口完成接入预留，可作为 V0 增强项但不替代 Web 主入口。

#### 功能验收

- 知识库问答支持答案、来源文档、引用片段展示。
- CRM 只读接入可返回客户、商机、联系人、跟进记录、已购产品或等价上下文。
- 客户分析输出结构化结果，包括客户背景、痛点、采购动机、风险点、推荐切入角度。
- 方案生成输出完整章节，不允许只有散点内容。
- pptx 生成使用公司标准模板，默认 12-18 页。
- docx 生成使用公司标准文档模板，包含封面、目录、章节、引用尾注或脚注。
- xlsx 生成使用固定表格模板，至少覆盖需求矩阵、功能清单、引用证据清单或报价/配置清单中的一种。
- 方案和 docx/pptx/xlsx 均可下载、可编辑、可重新生成。
- 用户能对问答、方案、交付物提交反馈。
- 飞书 Bot 至少完成事件回调、用户映射和 PAS QA Service 调用链路设计；如 V0 实现问答，只能复用 PAS 问答服务。

#### 质量验收

- 至少完成 50 个真实知识库问题验证。
- 核心问题可回答率不低于 80%。
- 方案中涉及产品能力的段落必须有知识库依据。
- pptx 不允许出现明显空页、标题缺失、大面积文字溢出。
- docx 不允许出现空章节、目录错乱、引用缺失。
- xlsx 不允许出现关键列缺失、表头错误、单元格严重溢出或公式错误。
- 客户背景页必须来自 CRM 或用户补充信息，不能由 AI 编造。
- 低置信、资料不足、引用不足时必须提示人工补充。

#### 安全验收

- 普通用户不能上传、删除、停用知识库资料。
- 普通用户只能查看自己的生成记录。
- 敏感合同、报价底线、客户隐私高风险资料不进入 V0 公共知识库。
- 所有 AI 生成方案和 docx/pptx/xlsx 必须提示“需人工审核后使用”。
- 系统记录问答、检索、生成、下载、反馈日志。
- 飞书 Bot 不允许直连 RAGFlow；未完成身份映射时只能访问公共知识库。

#### 上线验收

- 有全员可访问入口。
- 有 1 页使用说明。
- 有飞书 Bot 使用边界说明。
- 有管理员名单和反馈处理人。
- 有上线问题跟踪表。
- 完成至少 10 名销售/售前试用。
- 试用中至少跑通 5 个真实客户/商机场景。

### 2.3 V1 验收目标：稳定 MVP

V1 的验收目标是把 V0 的“能用”提升为“稳定、可维护、可持续运营”。核心验收点是知识库质量、引用可靠性、模板扩展、权限和运营闭环。

#### 业务验收

- 销售/售前可以稳定使用知识库、方案、多格式交付物三条主链路。
- 知识库管理员可以独立维护资料、查看解析状态、处理反馈。
- 售前负责人可以维护方案模板和 docx/pptx/xlsx 模板。
- 管理者可以看到知识库使用、方案生成、用户反馈的基础运营数据。

#### 功能验收

- 建立知识库质量门禁，覆盖 PDF、PPT、图片、扫描件。
- 支持文档解析状态、解析失败原因、重新解析。
- 支持知识标签：产品、行业、场景、竞品、资料类型。
- 支持至少 3 类方案模板：标准产品方案、客户痛点方案、竞品替换方案。
- 支持至少 3 套 pptx 模板或 20 个以上固定页面版式，并补齐 docx/xlsx 模板管理。
- 支持单页 pptx、单份 docx、单张 xlsx 重新生成。
- 支持文档级权限、模板维护权限、生成记录权限。
- 支持反馈分类、处理状态和知识补充闭环。

#### 质量验收

- 建立不少于 100 个知识库回归问题。
- 核心知识问题可回答率目标不低于 85%。
- 有引用的回答中，引用可追溯率目标不低于 90%。
- 方案完整性评分、引用覆盖率、客户匹配度形成可记录指标。
- 多格式交付物自动检查覆盖 pptx 空页/缺标题/文字超长、docx 空章节/目录错乱、xlsx 表头/关键列缺失、引用缺失。
- 至少完成 5 个历史真实方案回测。
- 至少完成 5 份真实客户多格式交付物生成验收。

#### 安全验收

- 权限至少支持部门、角色、管理员三类控制。
- 文档可设置可见范围。
- AI 检索前必须按权限过滤资料。
- AI 输出后必须做基础敏感词和越权内容检查。
- 管理员能追溯用户问题、命中文档、生成内容和下载记录。

#### 上线验收

- V0 遗留 P0 问题全部关闭。
- V1 新增功能有验收记录。
- 知识库管理员具备日常维护 SOP。
- 售前团队确认模板可用于真实工作。
- 形成每周知识库质量报告和反馈处理机制。

### 2.4 V2 验收目标：核心业务闭环版

V2 的验收目标是从“生成工具”升级为“售前业务闭环”。验收重点是 M2/M3/M4/M5/M12/M15 等模块是否能进入真实工作流程。

#### 业务验收

- 销售能把文字/语音/CRM 信息整理成结构化商机。
- 商机确认后能保存或回写到外部 CRM。
- 售前会议后能生成纪要，并能进一步生成方案草稿。
- 售后人员能使用售后知识库问答和维保提醒。
- 法务/财务能使用合同审核和风险提示。
- 渠道场景能读取渠道客户、渠道价格或报备信息。
- 客情系统能给出客户画像、动态信号和销售行动建议。

#### 功能验收

- M2 商机获取：文本商机整理、一键保存、招投标信息提醒至少一条链路可用。
- M3 会议场景：会议纪要、需求提取、会后方案生成可用。
- M4 合同与财务：合同条款审核、风险点标注、修改建议可用。
- M5 售后维保：售后知识专区、售后问答、维保提醒可用。
- M12 渠道管理：渠道信息、渠道报价、渠道报备至少支持只读接入。
- M15 客情系统：画像、痛点、采购窗口、竞品介入、行动建议可用。
- M11 数据埋点：问答、方案、docx/pptx/xlsx、商机、会议、售后、合同使用数据可采集。

#### 质量验收

- 至少完成 10 个真实商机场景验证。
- 至少完成 5 个真实会议纪要和会后方案验证。
- 至少完成 5 份合同审核样本验证。
- 售后问答核心问题可回答率目标不低于 80%。
- 客情分析必须能说明依据来源：CRM、活动记录、知识库或人工补充。
- 招投标/商机提醒必须有去重和人工确认机制。

#### 安全验收

- 不同角色只能访问对应业务数据。
- 商机、合同、售后、渠道数据必须记录来源系统。
- 写回 CRM 或业务系统的动作必须人工确认。
- 合同和报价相关信息必须按权限控制。
- 所有跨系统调用必须记录审计日志。

#### 上线验收

- 每个进入 V2 的模块至少有一个真实业务负责人签收。
- 每个业务模块有 3-5 个标准演示场景。
- 每个业务模块有失败降级路径。
- V1 的知识库、方案、多格式交付物主链路稳定运行，不因 V2 扩展下降。

### 2.5 V3 验收目标：平台化版本

V3 的验收目标是规模化、平台化和管理闭环。它不是继续堆功能，而是让 PAS 可以跨渠道、跨角色、跨产品、跨团队稳定运行。

#### 业务验收

- 用户可以通过 Web、飞书/企微等至少两个渠道使用核心能力。
- 管理层可以通过驾驶舱查看销售、售前、知识库、方案、渠道等指标。
- Agent/Skill 可以按审核流程入库和启用。
- 不同产品线可以通过产品注册表接入售前能力。
- CIP 客情能力能支撑持续跟进、预警和行动建议。

#### 功能验收

- M8 多渠道：至少完成一个企业 IM 渠道接入。
- M10 Agent 与编排：支持 Agent 注册、Skill 审核、简单工作流。
- M11 经营驾驶舱：支持销售漏斗、方案生成、知识库使用、反馈质量等核心指标。
- M13 可扩展：产品注册表、模板注册、Webhook/API 版本策略可用。
- M15 CIP 深化：客户画像、动态信号、预警、行动建议形成闭环。
- 多租户或多组织隔离字段预留完成。

#### 质量验收

- 核心链路可用性目标不低于 99%。
- 关键异步任务必须可重试、可追踪、可恢复。
- 经营指标每天自动更新。
- Agent/Skill 上线前必须经过安全扫描、功能验证、业务匹配、人工审批。
- 多渠道消息和 Web 端记录能统一归档。

#### 安全验收

- 多渠道入口继承同一套用户身份和权限。
- Agent/Skill 执行必须有权限矩阵和审计日志。
- 管理驾驶舱不展示用户无权访问的明细数据。
- API/Webhook 有版本、鉴权、限流和调用审计。
- 商业化/多租户预留不能破坏现有单组织使用。

#### 上线验收

- V0-V2 主链路稳定运行。
- 至少两个业务团队持续使用。
- 管理层驾驶舱指标经过业务确认。
- Agent/Skill 审核流程有实际样例跑通。
- 平台扩展机制能接入一个非 IP-Guard 产品或代理产品样例。

### 2.6 版本通用验收材料

每个版本上线前必须沉淀以下材料：

- 版本范围说明。
- 已完成任务清单。
- 未完成和暂缓清单。
- 验收用例和执行结果。
- 真实业务样例输出。
- 风险和遗留问题。
- 上线回滚方案。
- 管理员和业务负责人签收记录。

#### 3.17.4 PAS 自有四容器部署拓扑

V0-V3 基线采用 Sword PMS 的轻量容器思路：PAS 自有运行容器只保留四类，不因业务模块增加而拆更多服务容器。

| 容器 | 角色 | 端口 | 说明 |
|---|---|---|---|
| `pas-frontend` | Nginx 静态 SPA + 反向代理 | `80/443`，可映射到内网端口 | 唯一 Web 入口，代理 `/api` 到 backend |
| `pas-backend` | NestJS API + 全部业务模块 + BullMQ worker/scheduler | `3000/3001` 内网 | Auth、CRM、RAGFlow adapter、QA、Proposal、Export、Feedback、Feishu Integration 全部封装在 backend 内 |
| `pas-postgres` | PostgreSQL 16 | `5432` 内网 | 用户、权限、客户上下文、生成记录、模板、反馈、审计、导出记录 |
| `pas-redis` | Redis 7 | `6379` 内网 | 缓存、会话、BullMQ 队列、限流辅助 |

RAGFlow 不算 PAS 自有容器。RAGFlow 及其 ES/MySQL/MinIO/Valkey 等依赖作为外部 RAG 能力栈，通过 `RAGFLOW_BASE_URL`、`RAGFLOW_API_KEY`、`KB_ID` 接入。

```text
Browser
  -> pas-frontend / Nginx static SPA
      -> /api reverse proxy
        -> pas-backend / NestJS API
            -> pas-postgres
            -> pas-redis
            -> external RAGFlow stack
            -> external CRM
            -> LLM Provider
            -> FilesModule local backend volume or external S3/OSS/MinIO
```

Backend 容器内部包含所有模块和后台任务：

- API controller / service。
- RAGFlow adapter、CRM adapter、LLM adapter。
- QA、CustomerContext、Proposal、ExportModule、Feedback、FeishuIntegration。
- BullMQ processor、定时任务、质量回归任务。
- 文件生成和导出任务。

V0-V3 基线不拆独立 worker 容器，不新增 PAS 自有 MinIO 容器。若 V3 之后出现高并发或独立交付要求，再评估 worker 或对象存储独立化，但不作为当前任务书默认范围。

#### 3.17.5 前端部署边界

前端采用新 PAS 项目的确认方案：

```text
Vite + React + TypeScript + Ant Design
Nginx 静态 SPA 部署
```

边界要求：

- 前端不持有 RAGFlow、CRM、LLM 的密钥。
- 前端通过 NestJS API 访问所有外部系统。
- `index.html` 禁缓存，`assets` 长缓存。
- `/api` 由 `pas-frontend` 的 Nginx 反向代理到 `pas-backend`，`pas-backend` 不作为公网入口。
- 生成进度优先用 SSE；需要双向通信时再引入 WebSocket。

#### 3.17.6 文件存储策略

PAS 对齐 Sword PMS 的 `MinIO Client` 技术选择，但只把它作为 `backend` 内的 SDK/adapter，不把 `MinIO Server` 作为 PAS 默认容器或 backend 内嵌进程。

V0 默认配置：

```text
FILE_STORAGE_DRIVER=local
FILE_STORAGE_ROOT=/data/pas-files
```

`pas-backend` 挂载持久化 volume：

```text
pas_backend_files:/data/pas-files
```

后续切换外部对象存储时，仅替换 `FilesModule` driver：

```text
FILE_STORAGE_DRIVER=minio
MINIO_ENDPOINT=...
MINIO_BUCKET=pas-files
```

业务模块不得直接调用 `MinIO Client`。统一调用边界如下：

```text
ExportModule -> FilesModule.save()
KnowledgeModule -> FilesModule.upload()
ProposalModule -> FilesModule.getDownloadUrl()
```

这个设计保留 4 容器部署、V0 本地读写性能和后续切换 S3/OSS/MinIO 的扩展能力。

#### 3.17.7 不采用 Sword PMS 的部分

| Sword PMS 能力 | PAS 是否采用 | 原因 |
|---|---|---|
| 59 个业务模块的一体化 ERP/PMS 模型 | 不采用 | PAS 聚焦 AI 售前，不做 ERP 全业务系统 |
| 117 张表级别的 CRM/财务/库存/工单全套模型 | 不采用 | V0/V1 只保留客户上下文、生成记录、模板、反馈、权限等必要模型 |
| bytenode 代码保护 | 不采用 | 内部系统优先可调试、可维护 |
| License 机器绑定 | 不采用 | 非外售安装包，不需要首期 License enforcement |
| 无抽象的本地文件上传存储作为主模式 | 不采用 | V0 可用 `FilesModule local driver + backend volume`，但业务模块不得绕过 FilesModule 直接读写本地路径 |
| Node.js 24 | 不采用 | 改用 Node.js 22 LTS |

| 字段 | 内容 |

统一导出接口：

```text
POST /api/internal/exports
GET /api/internal/exports/{exportJobId}
GET /api/internal/exports/{exportJobId}/files/{format}/download
```

`ExportPackage` 标准数据包：
- `sourceType`：`proposal`。
- `sourceId`：方案 ID。
- `customerContext`：客户上下文。
- `customerAnalysis`：客户分析。
- `proposalSections`：方案章节。
- `evidenceChunks`：引用证据。
- `formatTargets`：`docx`、`pptx`、`xlsx`。
- `templateId`：模板 ID。

V0 xlsx 表结构：
- 需求矩阵：需求、痛点、对应能力、优先级、引用来源。
- 功能清单：模块、功能点、适用场景、交付阶段、备注。
- 引用证据清单：文档名、页码/位置、引用片段、关联章节。
- 报价/配置清单预留：产品项、数量、单位、备注；V0 不自动生成价格。

生成原则：
- 三类文件都走 `ExportModule`，不让业务模块各自生成文件。
- docx 使用带占位符的公司 `.docx` 模板，通过 `docxtemplater` 填充，保证可编辑和可人工修订。
- pptx 使用带占位符/版式页的公司 `.pptx` 母版，通过 `docxtemplater` 或 `pptx-automizer` 填充/组装，不能用代码重画母版作为 V0 主链路。
- xlsx 使用固定表头、列宽、冻结行和数据校验，不能输出自由表格。
- LLM 只生成标题、要点、讲稿、备注、表格行内容和引用说明。
- 输出文件必须通过 `FilesModule` 写入外部对象存储或 backend 挂载卷，并记录导出任务、下载日志和失败原因。

## 6. V0 验收标准

### 6.1 功能验收

- 用户能登录 PAS。
- 用户能搜索或选择 CRM 客户/商机。
- 系统能生成客户情况分析。
- 用户能向知识库提问。
- 问答能返回引用来源。
- 用户能一键生成 IP-Guard 方案草稿。
- 用户能一键生成 docx、pptx、xlsx。
- 用户能下载方案和 docx/pptx/xlsx。
- 用户能提交反馈。

### 6.2 质量验收

- 50 个真实知识库问题中，核心问题可回答率不低于 80%。
- 方案生成结果必须包含完整章节。
- 产品能力描述必须可追溯到知识库引用。
- pptx 默认 12-18 页。
- pptx 不出现明显空页。
- pptx 不出现大面积文字溢出。
- docx 不出现空章节、目录错乱、引用缺失。
- xlsx 不出现关键列缺失、表头错误、公式错误。
- 客户背景页必须来自 CRM 或用户补充信息。
- AI 生成材料必须显示人工审核提示。

### 6.3 性能验收

- 知识库问答单次响应建议不超过 30 秒。
- 方案生成建议不超过 10 分钟。
- 多格式交付物生成建议不超过 30 分钟。
- 失败任务必须有错误提示和重试入口。

## 7. V1 详细任务拆解

### 7.1 V1 目标

V1 的目标是把 V0 从“能用”提升到“稳定可运营”。重点不是大规模新增业务模块，而是把知识库、引用、方案、多格式交付物、权限、反馈、运营后台打磨到可以长期使用。

V1 核心链路如下：

```text
V0 使用数据
  -> 汇总反馈和失败样例
  -> 补知识库质量门禁
  -> 增强引用和权限
  -> 扩展方案/docx/pptx/xlsx 模板
  -> 建立质量评分、运营后台和知识块审核发布
  -> 形成稳定 MVP
```

## 11. 关键风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| RAGFlow 解析 PDF/PPT/图片质量不稳定 | 回答和引用不准 | V0 先人工筛选核心资料，V1 建解析质量门禁 |
| 旧 E0-E5 代码被误当作新项目基线 | 重新引入旧架构包袱，影响 V0 速度和边界 | 明确只迁移 RAGFlow 资产，旧应用代码只读参考，不进入新项目代码库 |
| CRM 接口不完整或延迟 | 客户分析不完整 | V0 支持手工补充客户信息 |
| 多格式交付物生成效果不达预期 | 用户感知差 | 固定 docx 结构、pptx 版式和 xlsx 表格模板，不做自由设计 |
| 知识库资料不足 | 方案空洞 | 先导入核心产品、历史方案、FAQ、竞品材料 |
| 用户误把 AI 输出直接外发 | 商务风险 | 明确人工审核提示和下载页水印/提示 |
| 一周范围过大 | 交付失败 | V0 只保留三个入口：问知识库、生成方案、生成交付物 |
| 权限过复杂拖慢 V0 | 无法按时上线 | V0 最小权限，敏感资料暂不导入 |

## 16. 结论

PAS 可以推翻原来的完整建设节奏，但不能推翻核心方法论。新的执行方式应该是：

```text
V0 先让全公司用起来
V1 把知识、引用、方案、多格式交付物质量打磨稳定
V2 扩展销售/售前业务闭环
V3 平台化、智能化、商业化
```

V0 的核心不是“完整 PAS”，而是“可实际使用的售前生产力工具”。只要 V0 把知识库、CRM 客户上下文、客户分析、方案草稿、docx/pptx/xlsx 生成跑通，后续模块就有真实使用数据和反馈可以继续迭代。
